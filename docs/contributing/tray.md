# Tray app ŌĆö contributor notes

User-facing install/usage docs live in [`docs/tray.md`](../tray.md). This page is the SwiftUI / macOS gotcha collection ŌĆö read it before editing `src/Mailvec.Tray/`.

## Architecture

- **Plain REST, not MCP framing.** The tray talks to `/tray/status`, `/tray/system`, `/tray/search`, `/tray/control`, `/tray/folders`, `/tray/attachment` on the same Kestrel host (`127.0.0.1:3333`) as MCP. These endpoints are deliberately *not* MCP tools ŌĆö they need session-less access, fields MCP tools don't return (services, progress, sparkline), and a flatter shape than the LLM-facing tools. Registered before `MapMcp()` in [Program.cs::RunHttp](../../src/Mailvec.Mcp/Program.cs); routes live in [TrayEndpoints.cs](../../src/Mailvec.Mcp/Tray/TrayEndpoints.cs).
- **Wire-format contract.** The C# records in [TrayModels.cs](../../src/Mailvec.Core/Tray/TrayModels.cs) and the Swift structs in [Models.swift](../../src/Mailvec.Tray/Sources/Models.swift) are paired one-for-one. Field names use camelCase via the default System.Text.Json options. Rename either side without the other and Swift silently decodes null ŌĆö there's no compile-time check across the language boundary. Bump field names in lockstep.
- **In-memory ring buffer lives in the MCP process, not the embedder.** [TrayEventRecorder](../../src/Mailvec.Core/Tray/TrayEventRecorder.cs) is a `BackgroundService` registered only in the MCP server. It samples the DB once a minute and keeps a 30-bucket deltas array. *Why not in the embedder*: the embedder runs in a separate process; sharing heap would require IPC, and sampling the DB the MCP server already reads is simpler. Cold-start has 30 zero buckets; refills over the next half-hour. If you ever need true per-event granularity, add a small `events` table ŌĆö don't try to read the embedder's memory.
- **`launchctl print` parsing is brittle but stable enough.** [LaunchdInspector](../../src/Mailvec.Core/Tray/LaunchdInspector.cs) shells out to `launchctl print gui/<uid>/<label>` and parses `state`, `pid`, `last exit code`, `runs` out of the text. Apple's XPC C API for this is private; every macOS menu-bar app does the same. The parser is unit-tested in [LaunchdInspectorTests.cs](../../tests/Mailvec.Core.Tests/Tray/LaunchdInspectorTests.cs) ŌĆö if a future macOS changes the output, the parser regresses there first.
- **Pause/Resume acts on indexer + embedder together, never mbsync.** [`TrayModel.pauseServices`](../../src/Mailvec.Tray/Sources/TrayModel.swift) calls `bootout` on both .NET services so the user can stop ingestion + embedding without halting mail delivery. mbsync stays on its timer; pausing it would mean editing the plist's `StartInterval` and reloading, which the tray doesn't do.

## Build chain

- **No App Sandbox.** The tray reads `~/Library/Logs/Mailvec/`, shells out to `launchctl`, and uses AppleScript to spawn Terminal for `mailvec doctor` / `reindex`. App Sandbox would block all three. Hardened Runtime is on; the user installs the .app explicitly, so this is acceptable for a local-only tool.
- **XcodeGen, not a checked-in `.xcodeproj`.** [`project.yml`](../../src/Mailvec.Tray/project.yml) is the source of truth; the generated `.xcodeproj` is gitignored. Anyone touching the tray runs `xcodegen generate` first (or just `ops/build-tray.sh`, which does it for you). Don't commit the `.xcodeproj` even temporarily ŌĆö it's a binary-ish pbxproj that doesn't diff.
- **No SPM dependencies.** The tray previously pulled in [soffes/HotKey](https://github.com/soffes/HotKey) for a global ŌīśŌć¦M shortcut. Carbon hotkey registration worked, but SwiftUI's `MenuBarExtra` has no public API to open the popover programmatically ŌĆö and the KVC-into-`NSStatusBarWindow` workaround used by [orchetect/MenuBarExtraAccess](https://github.com/orchetect/MenuBarExtraAccess) wasn't stable across SwiftUI versions in our environment. The hotkey would fire and mutate state without surfacing UI, so the whole feature was removed cleanly. Reinstate if Apple ever ships an `isPresented:` binding on `MenuBarExtra`.
- **Local notifications need at least ad-hoc signing.** `UNUserNotificationCenter.requestAuthorization` rejects fully-unsigned bundles with `UNErrorDomain Code=1` (notifications not allowed) and doesn't register the bundle with Notification Center at all. [`ops/build-tray.sh`](../../ops/build-tray.sh) sets `CODE_SIGN_IDENTITY="-"` + `CODE_SIGN_STYLE=Manual` to force ad-hoc signing ŌĆö no Developer Team needed but the signature must exist. Verify with `codesign -dv /Applications/Mailvec.Tray.app` (look for `Signature=adhoc`).
- **App icon ships as an `Assets.xcassets/AppIcon.appiconset`.** The bundle's icon file (Notification Center banner, Finder icon, Cmd-Tab) is built from the 128px source at [`assets/mailvec-128.png`](../../assets/mailvec-128.png) via `sips` resizing in `xcodebuild`'s asset-catalog compiler. Asset catalog **must** be listed under `sources:` in [`project.yml`](../../src/Mailvec.Tray/project.yml), not `resources:` ŌĆö the latter copies the directory verbatim instead of compiling it into the `Assets.car` macOS reads from. Don't undo this.

## SwiftUI quirks

- **Date decoding is custom, not `.iso8601`.** The .NET server emits `DateTimeOffset.ToString("O")` with fractional seconds (e.g. `2026-05-14T19:03:42.909205+00:00`). Foundation's `.iso8601` strategy doesn't accept fractional seconds. [`MailvecClient`](../../src/Mailvec.Tray/Sources/MailvecClient.swift) uses a custom strategy with two `ISO8601DateFormatter` instances (with + without fractional seconds) for tolerance.
- **`Section("Title") { content } footer: { ... }` is NOT valid SwiftUI.** The matching Section initializer requires `Section { content } header: { Text("Title") } footer: { ... }`. The handoff used the shorter form; every site got rewritten. If you add a new section with both a title and a footer, use the long form.
- **Preferences is a `WindowGroup`, not the SwiftUI `Settings` scene.** The `Settings` scene requires the app's activation policy to be `.regular` to surface ŌĆö for `LSUIElement` accessory apps that means bouncing between `.accessory` and `.regular` every time the user opens Prefs, which after 3-4 cycles reliably corrupts `NSStatusItem` state and leaves the menu-bar icon visible but unresponsive to clicks. [`MailvecTrayApp`](../../src/Mailvec.Tray/Sources/MailvecTrayApp.swift) uses `WindowGroup(id: "preferences")` opened via `\.openWindow` instead ŌĆö works from `.accessory` mode permanently with no policy flips. Don't switch back to `Settings`.

## External-source-of-truth dependencies

- **Claude Desktop's MCPB install dir changed names.** Older builds use `~/Library/Application Support/Claude/Connectors/`, current builds use `~/Library/Application Support/Claude/Claude Extensions/`. [`TraySystemService.McpbInstalled`](../../src/Mailvec.Core/Tray/TraySystemService.cs) checks both, preferring the new name. If a future version renames again, add the new path to that array.
- **`/tray/system` parses three external sources of truth that the .NET process doesn't otherwise read.** Code lives in [TraySystemService.cs](../../src/Mailvec.Core/Tray/TraySystemService.cs): (1) `~/.mbsyncrc` for `Host` + `User` directives, (2) `~/Library/LaunchAgents/com.mailvec.mbsync.plist`'s `StartInterval` to display the schedule, (3) a `VERSION` sidecar next to `vec0.dylib` (written by [`ops/fetch-sqlite-vec.sh`](../../ops/fetch-sqlite-vec.sh)) for the sqlite-vec version. None of these are watched ŌĆö they're re-read on every `/tray/system` request. Editing the plist and re-running `ops/install.sh` is enough for the schedule to update in the UI on the next poll.

## CLI shim integration

- **The `mailvec` CLI shim path is duplicated across three tray callsites.** [`CliRunner.swift`](../../src/Mailvec.Tray/Sources/CliRunner.swift) is the single helper that resolves `~/.local/bin/mailvec` (preferred) ŌåÆ `/usr/local/bin/mailvec` (fallback) and spawns Terminal via AppleScript. Don't re-roll the lookup in new buttons ŌĆö call `CliRunner.runInTerminal(["doctor"])` etc. and the resolver stays in one place.

## Tests + coverage

- **Wire-format contract tests.** [`Mailvec.Tray.Tests`](../../src/Mailvec.Tray/Tests) is an XCTest target wired up via [`project.yml`](../../src/Mailvec.Tray/project.yml). It round-trips hand-written fixture JSON through the production `JSONDecoder` to catch field renames between [`TrayModels.cs`](../../src/Mailvec.Core/Tray/TrayModels.cs) and [`Models.swift`](../../src/Mailvec.Tray/Sources/Models.swift). The decoder + fractional-second normaliser live in [`WireDecoder.swift`](../../src/Mailvec.Tray/Sources/WireDecoder.swift) (not on `MailvecClient`) so the test target imports the same instance shape the app uses. Run standalone with `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer xcodebuild test -project Mailvec.Tray.xcodeproj -scheme Mailvec.Tray -destination "platform=macOS"`.
- **Coverage merges into the .NET ReportGenerator output.** [`ops/coverage.sh`](../../ops/coverage.sh) has a guarded block that runs the Swift tests with `-enableCodeCoverage` (already on in the scheme), converts the resulting `.xcresult` to Cobertura via [`xcresultparser`](https://github.com/a7ex/xcresultparser), and drops it under `coverage/raw/` alongside the .NET reports. ReportGenerator picks it up automatically; the tray appears as a `Mailvec.Tray` assembly in [`coverage/SummaryGithub.md`](../../coverage/SummaryGithub.md). Prereq: `brew install xcresultparser` plus Xcode.app present at `/Applications/Xcode.app` (Command Line Tools alone isn't enough ŌĆö `xcresultparser` shells out to `xcresulttool`, which only ships with Xcode). If either prereq is missing, the Swift step is skipped silently and the .NET report runs as before.
- **Why we sed-rewrite the cobertura XML.** xcresultparser derives Cobertura `package` and `class` names from the file's directory path, so the Swift coverage initially shows up as three sibling "assemblies" (`src.Mailvec.Tray.Sources`, `src.Mailvec.Tray.Sources.Components`, `src.Mailvec.Tray.Sources.Preferences`) with class names like `src.Mailvec.Tray.Sources.src/Mailvec.Tray/Sources/WireDecoder`. The post-process pass in `ops/coverage.sh` collapses them into a single `Mailvec.Tray` assembly with class names like `Mailvec.Tray.Sources.WireDecoder` ŌĆö matches the .NET naming convention so the per-file rows line up in the report.
